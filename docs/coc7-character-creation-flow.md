## 正式 CoC7 车卡流程

按规则和模板综合后，较稳定的流程应当是：

1. 先确定调查员概念。
   包括时代、地点、调查员的大致身份、是否贴合模组入口。规则书明确要求先和 KP 对齐模组前提，而不是先盲掷数值。

2. 决定职业。
   职业先于技能分配，因为职业会定义：
   - 本职技能点公式
   - 信用评级范围
   - 本职技能清单
   - 可选技能组与自由技能位

3. 生成属性。
   标准属性仍按 CoC7 常规方法生成：
   - STR / CON / POW / DEX / APP：`3D6*5`
   - SIZ / INT / EDU：`(2D6+6)*5`
   - Luck：`3D6*5`

4. 处理年龄修正。
   这是当前实现里最容易做偏的一步。正式规则不是把减值固定拆到某几个属性上，而是：
   - `15-19`：`STR+SIZ` 合计减 5，`EDU` 减 5，`Luck` 可重掷两次取最高
   - `20-39`：无属性减值
   - `40+`：按年龄段获得教育成长检定次数，并把 `STR/CON/DEX` 的总减值手动分配；`APP` 固定下降；`MOV` 随年龄段下降

   关键点：
   - `STR/CON/DEX` 或 `STR/SIZ` 的减值应由玩家决定如何分配
   - 教育成长检定是独立步骤，不应被写死成固定 EDU 加值
   - `MOV` 由年龄段自动修正即可

5. 计算衍生属性。
   至少包括：
   - HP
   - MP
   - SAN
   - MOV
   - DB
   - Build
   - 各属性半值 / 五分之一值

6. 分配职业技能点。
   以职业为中心完成：
   - 先确认职业强制技能
   - 再选完职业可选技能组
   - 再选职业允许的自由技能位
   - 最后分配职业技能点

   规则侧约束：
   - `信用评级` 必然属于职业技能体系
   - `兴趣点` 不应加到 `信用评级`
   - 本职技能一般不应超过八项，不足时可由“任意技能/个人或时代特长”补足

7. 分配兴趣技能点。
   当前项目按 `INT*2` 处理，这与常规 CoC7 角色卡逻辑一致。

8. 处理经历包。
   如果启用“有故事的调查员”可选规则，建议在第二章就完成：
   - 经历包选择
   - 年龄门槛检查
   - 经历包可选技能组勾选
   - 经历包技能点分配

   这样可以避免第三章再补背景信息时才发现技能总值或点池超限。

9. 写背景与关键链接。
   这一步不只是文本补完，还会反向决定角色的驱动力与导出卡片完整度。建议至少完成：
   - 个人描述
   - 思想与信念
   - 重要之人
   - 意义之地
   - 珍视之物
   - 特质
   - 关键链接

10. 处理财产、武器、装备。
   财务部分应由 `信用评级` 反推消费水平、现金、资产范围。武器应从武器库选择，不直接硬编码在前端。

## 对现有 App 的业务映射

当前五阶段结构可以保留，但建议按下面的规则理解：

- `序章`
  - 负责概念输入、时代、年龄、职业选择
  - 不应在这里提前做技能点分配

- `第一章`
  - 负责属性生成、年龄修正、教育成长检定、衍生值确认
  - 年龄修正必须保持“固定项自动、手动项显式分配”的模式

- `第二章`
  - 负责职业技能组选择、信用评级、本职技能点、兴趣点、经历包与经历包技能点分配
  - 这是最核心的规则阶段

- `第三章`
  - 负责背景、关键链接、财务、装备、武器
  - 这里只保留经历包摘要，不再承接经历包点数分配

- `终章`
  - 负责导出、模板映射校验、分享

## 当前项目已确认的逻辑风险

### 1. 年龄修正不能写成固定拆分

旧实现把 `40+` 的 `STR/CON/DEX` 总减值写成固定配比，例如 `2/2/1`、`4/3/3`。这不符合规则。

正确做法：

- 自动处理固定项：
  - `APP` 固定减值
  - `EDU` 的青年减值
  - `MOV` 自动减值
- 把体能减值留给玩家手动分配
- 仅给出软提醒，不强制锁死

### 2. 子类技能基础值必须跟着子类变

Excel 已经解析出了子类库，且不同子类的基础值并不总相同，例如：

- `格斗(斗殴)` 基础值和 `格斗(链锯)` 不同
- `射击(手枪)`、`射击(步枪/霰弹枪)`、`射击(机枪)` 基础值不同

如果前端只按大类基值计算，总值就会错。

### 3. 职业选择数据应优先信任 Excel 结构化结果

`data/occupations.from_excel.json` 已经提供了：

- 本职技能点公式
- 信用评级范围
- `mandatoryRefs`
- `choiceGroups`
- `freePickCount`

因此前端不应该再把 `skillText` 当成唯一真相来源去重新猜测。文本解析只能作为兜底。

### 4. 草稿恢复不能冲掉职业可选项选择结果

运行时重新加载职业数据时，如果重新套用职业但把 `groupPicks` 和 `freePicks` 清空，用户草稿就会失真。这属于典型业务状态回填错误。

## 本轮优化结论

本轮应坚持以下实现原则：

- 规则结构以 Excel 与规则书为准，文本猜测只作 fallback
- 错误时给出软提醒，不阻断玩家操作
- 导出层继续只写入输入格，不覆盖模板公式
- 技能子类一旦选定，基础值必须实时切换
- 年龄修正必须可追溯、可撤销、可重算

## 后续开发建议

- 如果后面要接入 DND 车卡器，不建议把规则逻辑继续写散在组件里；应把“规则引擎层”和“展示层”拆开
- CoC7 这里已经可以抽象成：
  - 属性生成规则
  - 年龄修正规则
  - 职业技能选择规则
  - 技能基值规则
  - 导出映射规则

未来新增规则系统时，优先复用这五类抽象，而不是复用具体界面组件。
